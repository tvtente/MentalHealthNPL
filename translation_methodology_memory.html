Memoria metodológica y técnica del proceso de traducción del dataset

Descripción del flujo de traducción automatizada, control de errores, rescate de textos largos y criterios metodológicos aplicados al corpus de salud mental

Propósito científico de esta sección

Esta memoria describe el procedimiento metodológico y técnico implementado para traducir al español un corpus textual de gran tamaño originalmente redactado en inglés y vinculado a salud mental. El objetivo fue construir una versión lingüísticamente operativa del dataset para fines de análisis exploratorio, clasificación automática y Procesamiento de Lenguaje Natural, preservando en la medida de lo posible el significado semántico, el tono emocional y el contexto discursivo de los textos originales.

Dado que el corpus contiene textos sensibles, extensos y heterogéneos, el flujo requirió mecanismos de control de calidad, tolerancia a fallos, reanudación de ejecución y rescate específico para filas problemáticas. Esta sección resume esas decisiones y deja constancia del criterio metodológico utilizado para la construcción del dataset traducido.

1. Modelo utilizado e infraestructura de ejecución

Identificación del modelo, forma de despliegue local y justificación operativa del entorno de traducción

¿Qué modelo se utilizó y cómo fue puesto en funcionamiento?

La traducción fue ejecutada mediante el modelo gemma3:4b, desplegado localmente con Ollama. El modelo puede descargarse desde la biblioteca oficial de Ollama y ejecutarse mediante su servicio local, lo que permite exponer un endpoint de inferencia dentro de la propia máquina de trabajo. Esta arquitectura hizo posible controlar el flujo desde scripts personalizados de Python, sin depender de llamadas continuas a servicios externos de traducción.

La elección de inferencia local respondió a criterios de coste, reproducibilidad, continuidad operativa y flexibilidad metodológica. En lugar de limitar el proceso a un traductor generalista o a una API comercial, se optó por un entorno que permitiera ajustar prompts, tamaños de lote, tiempos de espera, validación de formato y estrategias de recuperación ante fallos conforme avanzaba la complejidad del corpus.

El procedimiento implementado no se limita exclusivamente a la traducción hacia español. Dado que el flujo permite parametrizar tanto el idioma de origen como el idioma o variante lingüística de destino mediante códigos de locale, es posible generar versiones del corpus adaptadas a distintos idiomas soportados y, dentro de ellos, a diferentes regiones de uso, por ejemplo es-ES, es-AR, en-US o en-GB. Esta flexibilidad resulta metodológicamente relevante, ya que un modelo orientado a clasificación o análisis lingüístico puede requerir alineación no solo con una lengua general, sino también con convenciones léxicas, pragmáticas y culturales propias de la comunidad en la que se desea evaluar su rendimiento.

En este sentido, la traducción por locale no debe entenderse únicamente como una diferencia ortográfica o regional, sino como una estrategia de adaptación lingüística del corpus a contextos de uso específicos. Esto permite construir datasets derivados más coherentes con la variedad idiomática del entorno de aplicación, reduciendo el riesgo de introducir formas poco naturales o culturalmente desalineadas en fases posteriores de análisis, entrenamiento o evaluación.

2. Estructura operativa del flujo de traducción

Organización del archivo fuente, archivo de salida y criterio de continuidad sobre las celdas traducidas

¿Cómo se organizó el procesamiento para no perder avance?

El flujo principal se apoyó en un archivo fuente con el texto original y en un archivo de salida en formato CSV que acumuló progresivamente las traducciones en español. El sistema añadió una columna específica para la salida traducida y utilizó esa columna como criterio real de continuación: las filas con contenido se consideraron resueltas y las filas vacías se consideraron pendientes.

Este patrón hizo que la reanudación no dependiera del número absoluto de línea procesada, sino del estado efectivo del dataset de salida. Si la ejecución se interrumpía, bastaba con relanzar el proceso para que el sistema reconstruyera la lista de pendientes a partir de los espacios todavía vacíos en la columna traducida, sin reiniciar desde cero.

Para el caso de trabajo desarrollado en este proyecto, la ejecución controlada puede realizarse indicando explícitamente el archivo fuente, el archivo de salida, el locale de origen y destino, el tamaño de lote, la longitud de fragmento y el timeout. Un ejemplo representativo sería una configuración como source-locale = en-US, target-locale = es-ES, batch-size = 8, chunk-max-chars = 500 y request-timeout = 120, lo que permite mantener una traducción gradual, validada y reanudable sobre el corpus actual.

3. Segmentación por lotes y ajuste dinámico

Uso de lotes pequeños, reducción automática y adaptación al incremento de textos largos y complejos

¿Por qué fue necesario trabajar con lotes pequeños y variables?

A medida que el corpus avanzó, las filas más sencillas tendieron a resolverse primero y quedaron pendientes textos progresivamente más largos, narrativos y difíciles de serializar correctamente en la salida. Por esta razón, el tamaño de lote fue reducido hasta una configuración más estable de 8 filas por solicitud, especialmente adecuada para esta fase avanzada del dataset.

Además, el sistema fue diseñado para reducir automáticamente el lote cuando aparecían errores de formato, truncamiento, validación o timeout. La lógica aplicada fue una reducción por mitades, de modo que un lote podía pasar de 8 a 4, luego a 2 y finalmente a 1. Esto permitió distinguir entre fallos causados por el volumen conjunto del lote y fallos debidos a la longitud o complejidad de una sola fila.

4. Validación de salida y control de integridad

Verificación estricta del formato devuelto por el modelo y prevención de desalineaciones estructurales

¿Cómo se evitó incorporar respuestas dañadas al dataset?

El modelo fue instruido para devolver la salida en formato CSV con pares row_id, translation_es. Sin embargo, en la práctica se observaron respuestas truncadas, filas omitidas, identificadores equivocados y salidas semánticamente correctas pero estructuralmente inválidas. Para impedir que estas respuestas dañaran el dataset, se incorporó una capa de validación estricta antes de escribir cualquier resultado en el archivo de salida.

La validación comprobó que la respuesta incluyera exactamente los identificadores esperados, sin omisiones, duplicados ni row_ids ajenos al lote solicitado. Cuando esta condición no se cumplía, la salida no se integraba al archivo final, sino que el sistema la trataba como error de lote y reintentaba con un tamaño menor, preservando la integridad tabular del corpus traducido.

5. Gestión de bloqueos y tiempos de espera

Incorporación de timeout por petición y tratamiento de respuestas lentas como parte del circuito de recuperación

¿Qué se hizo cuando el modelo se bloqueaba o tardaba demasiado?

Durante la ejecución se detectaron situaciones donde el modelo local no fallaba explícitamente, pero quedaba excesivamente lento o aparentemente congelado sobre determinados textos. Para evitar que todo el proceso quedara suspendido indefinidamente, se introdujo un timeout por petición, de manera que cualquier solicitud que superara el tiempo configurado fuera tratada como un fallo controlado.

Esta decisión permitió incorporar los bloqueos dentro del mismo circuito de recuperación que los errores de formato o truncamiento. En términos operativos, un timeout no implicó abortar toda la traducción, sino reducir el lote o, si el problema persistía en tamaño mínimo, derivar la fila a una estrategia específica de rescate.

6. Tratamiento de filas problemáticas

Registro auxiliar de casos conflictivos y separación entre fallos de lote y fallos de longitud individual

¿Qué ocurrió con las filas que seguían fallando incluso en lote mínimo?

Cuando una fila continuaba fallando incluso con lote unitario, el problema dejó de interpretarse como un error de agrupación y pasó a considerarse un problema de longitud o complejidad individual de la celda. Para estos casos se creó un archivo auxiliar de problemáticas, donde se registraron el identificador de fila, la etiqueta contextual, la longitud del texto, el motivo del fallo, el texto fuente y la fecha de registro.

Esta separación permitió que el flujo principal siguiera avanzando sin perder trazabilidad sobre los casos difíciles. En lugar de bloquear toda la ejecución por unas pocas narraciones extensas o conflictivas, cada excepción quedó documentada y disponible para una intervención posterior o inmediata mediante otro mecanismo de rescate.

7. Recuperación por traducción fragmentada

División de textos largos en fragmentos manejables y recomposición posterior de la traducción final

¿Cómo se rescataron los textos demasiado largos para una sola respuesta?

Para las filas que seguían provocando truncamiento incluso en tamaño unitario, se implementó una estrategia de traducción fragmentada. El texto original se dividió en fragmentos de aproximadamente 500 caracteres, procurando respetar cortes razonables por frases y unidades discursivas. Cada fragmento se envió al modelo por separado y las traducciones parciales se recombinaron posteriormente en una sola salida final.

Esta recuperación por fragmentos fue integrada dentro del propio flujo principal. Es decir, una fila podía ser detectada como problemática, registrada en el archivo auxiliar y, a continuación, sometida de inmediato a un intento de rescate. Si la recomposición resultaba exitosa, la traducción se escribía en el archivo de salida y la fila se eliminaba de la cola de problemáticas; si no, quedaba pendiente para una segunda pasada específica sobre textos largos.

8. Justificación metodológica para textos sensibles

Razones para utilizar traducción local y prompts de traslado lingüístico en un corpus sobre salud mental

¿Por qué este enfoque ayudó a evitar negaciones o bloqueos por contenido sensible?

El corpus de origen contiene referencias a ideación suicida, trauma, desesperanza, culpa, aislamiento, angustia y otras expresiones intensas de sufrimiento psicológico. En entornos de traducción automatizada, este tipo de contenido puede activar respuestas conservadoras, interrupciones o reformulaciones improcedentes si el modelo interpreta la solicitud como una interacción asistencial. Para mitigar ese riesgo, el flujo fue planteado explícitamente como una tarea de traducción de corpus y no como un sistema de respuesta al hablante.

En consecuencia, los prompts se formularon para exigir un traslado fiel, neutral y no estigmatizante del contenido, evitando solicitar consejo, interpretación clínica o reformulación moralizante. El uso de inferencia local reforzó este control metodológico, ya que permitió iterar sobre el prompt y sobre la estrategia de procesamiento cuando el material sensible producía salidas truncadas o inconsistentes.

9. Trazabilidad, reanudación y límites del resultado

Registro de avance, continuidad entre ejecuciones y delimitación interpretativa del dataset traducido

¿Qué garantías metodológicas ofrece el procedimiento y cuáles son sus límites?

El proceso fue diseñado como un sistema trazable y reanudable. El archivo de salida, el checkpoint de avance y la cola de problemáticas permitieron auditar tanto la continuidad del procesamiento como la naturaleza de los errores encontrados. Desde un punto de vista operativo, esto hizo posible detener y relanzar la ejecución sin pérdida del trabajo consolidado y sin necesidad de recomenzar desde el inicio del corpus.

No obstante, el dataset traducido resultante debe entenderse como una construcción lingüística y computacional, no como una edición clínicamente validada. Aunque se introdujeron mecanismos para preservar fidelidad semántica, tono emocional y prudencia terminológica, toda traducción automática implica desplazamientos interpretativos potenciales. Su principal valor reside en ofrecer una base operativa para análisis automatizado en español, acompañada de una metodología transparente sobre cómo fue construida y bajo qué restricciones técnicas se desarrolló.

Criterio metodológico

La principal aportación de este procedimiento no reside únicamente en la generación de una columna traducida, sino en la construcción de un flujo robusto capaz de distinguir entre errores de lote, errores de longitud individual, bloqueos por tiempo y textos que requieren una recuperación fragmentada. Esta arquitectura fue necesaria para sostener la traducción de un corpus amplio, sensible y técnicamente exigente sin comprometer la integridad del dataset ni perder la trazabilidad de las excepciones.

En consecuencia, el dataset traducido debe leerse junto con su proceso de construcción. La validez técnica del recurso no depende solo del texto final publicado, sino también del método utilizado para producirlo, de las salvaguardas adoptadas frente a respuestas truncadas y del circuito de rescate que permitió conservar registros que de otro modo habrían quedado excluidos.

Asimismo, los scripts necesarios para esta traducción controlada, incluidos el flujo principal por lotes y el procedimiento auxiliar de rescate por fragmentos para textos problemáticos, quedan disponibles junto con el repositorio o material adjunto del proyecto. Esto favorece la reproducibilidad del proceso, la auditoría metodológica y la posibilidad de adaptar el mismo pipeline a otros idiomas, variantes regionales o contextos de evaluación.

El pipeline fue diseñado con parametrización explícita de locale, de modo que el idioma de origen y la variedad lingüística de destino puedan definirse mediante códigos normalizados como en-US, en-GB, es-ES o es-AR. Desde un punto de vista técnico, esta decisión permite desacoplar el procedimiento de una única configuración idiomática y reutilizarlo en escenarios donde la evaluación de modelos requiera alineación con una comunidad lingüística, un dialecto o una convención regional específica.

En este marco, la traducción por locale no se considera una simple preferencia ortográfica, sino una forma de control metodológico sobre la variedad de salida esperada. Esta distinción resulta importante porque una traducción culturalmente desalineada puede introducir sesgos léxicos, registros poco naturales o formulaciones pragmáticamente ajenas al entorno donde posteriormente se entrene, evalúe o interprete el modelo. Por ello, el flujo permite generar columnas de traducción diferenciadas por locale dentro del mismo archivo de salida, conservando variantes independientes y auditables.

En el caso concreto desarrollado en este proyecto, la transición hacia la nueva columna dependiente de locale puede realizarse primero como una migración única para preservar el trabajo acumulado previamente bajo una columna genérica. Una vez completado ese paso, el flujo normal puede continuar utilizando únicamente la parametrización por locale, sin necesidad de mantener instrucciones de renombrado en ejecuciones posteriores.

Un ejemplo de migración inicial para trasladar el contenido existente desde statement_es hacia la nueva columna statement_es_es es el siguiente: 

cd /home/tvt/MEGA/GIT/ART/NPL
source venv/bin/activate
python mental_health_translation_agent.py \
  --input "Combined Data.csv" \
  --provider ollama \
  --model gemma3:4b \
  --source-locale en-US \
  --target-locale es-ES \
  --rename-translated-col-from statement_es \
  --batch-size 8 \
  --min-batch-size 1 \
  --chunk-max-chars 500 \
  --request-timeout 120 \
  --skip-unrecoverable

Tras esa migración inicial, la ejecución reproducible del flujo principal puede mantenerse ya en su forma estable, donde la columna de salida queda determinada exclusivamente por target-locale = es-ES: 

cd /home/tvt/MEGA/GIT/ART/NPL
source venv/bin/activate
python mental_health_translation_agent.py \
  --input "Combined Data.csv" \
  --provider ollama \
  --model gemma3:4b \
  --source-locale en-US \
  --target-locale es-ES \
  --batch-size 8 \
  --min-batch-size 1 \
  --chunk-max-chars 500 \
  --request-timeout 120 \
  --skip-unrecoverable
